Por que QR Code no Pix?

O Pix tem várias formas de iniciar um pagamento: por chave Pix, por dados bancários, por aproximação NFC. O QR Code é, na prática, a forma mais utilizada em contextos de cobrança comercial — e a mais rica tecnicamente.

💡
A ideia central Um QR Code Pix é um envelope padronizado que carrega — ou aponta para — todas as informações de uma transação. O pagador nunca precisa digitar nada: basta apontar a câmera.

As três iniciações por QR Code que a regulação define

📱 Câmera lê QR
🔍 App processa payload
🏛️ Consulta DICT
✅ Pagador confirma
⚡ SPI liquida

O Art. 6, II da Res. BCB 1/2020 torna obrigatório que todo PSP (banco, fintech, etc.) disponibilize ao pagador a opção de iniciar um Pix via leitura de QR Code. Não é opcional — é mandatório do regulador.

🎯 Para o recebedor

Gerar um QR Code é o jeito mais simples de cobrar: basta compartilhar uma imagem. Sem precisar falar ISPB, agência, conta, dígito verificador.

🎯 Para o pagador

Apontar câmera = todos os dados preenchidos automaticamente. O app ainda mostra o nome do recebedor para confirmar antes de pagar.

🎯 Para o PSP

QR Code Dinâmico permite automação total da cobrança via API Pix: emissão, consulta de status, devolução, conciliação — tudo programático.

O padrão BR Code (base técnica)

O QR Code Pix não é um QR Code qualquer. Ele segue o BR Code, padrão brasileiro baseado na especificação internacional EMV-QRCPS (Europay, Mastercard, Visa — QR Code Payment Specification).

📐
Dois modos EMV — só um é o Pix O EMV define MPM (Merchant Presented Mode, QR exibido pelo recebedor) e CPM (Consumer Presented Mode, QR exibido pelo pagador). O Pix usa exclusivamente o MPM — o estabelecimento/recebedor exibe o código, o pagador escaneia.

O identificador global do Pix (GUI)

Todo QR Code Pix tem um campo que o identifica como sendo do arranjo Pix, e não de outro sistema de pagamento:

// Campo ID 00 dentro do bloco de Merchant Account Information (IDs 26 a 51) GUI = "br.gov.bcb.pix" // 14 chars, case-insensitive

Esse GUI (Globally Unique Identifier) é a assinatura digital do Pix dentro do BR Code. Qualquer app que implemente o padrão sabe que, ao encontrar esse identificador, está diante de um pagamento Pix.

Estrutura de campos do BR Code

📐
Formato TLV — Tag · Length · Value Cada campo é codificado como: ID (2 chars) + Comprimento (2 chars, zero-padded) + Valor. O comprimento indica o tamanho exato do valor — sem padding, sem espaços. Se um campo opcional estiver vazio, omite-se o campo inteiro.
Exemplo: nome "João" → 5904João (ID=59, comprimento=04, valor=João).
ID Campo EMV Obrig. Tipo Tamanho Observação / Valores aceitos Exemplo
00 Payload Format Indicator Sim String fixo 2 (fixo) Sempre "01" — versão do padrão EMV 000201
01 Point of Initiation Method Cond. Numérico 2 (fixo) 11 = uso único (QR dinâmico de cobrança); 12 = reutilizável (QR estático). Se ausente, tratado como reutilizável. 010211
26–51 Merchant Account Information Sim Estrutura aninhada 23–99 (total do bloco) Contém subcampos: 00=GUI (br.gov.bcb.pix), 01=chave Pix (estático) ou 25=URL (dinâmico). O bloco inteiro não pode ultrapassar 99 chars. 2658 0014br.gov.bcb.pix 0136{uuid}
52 Merchant Category Code (MCC) Sim Numérico 4 (fixo) Sempre 0000 para o Pix (categoria não especificada) 52040000
53 Transaction Currency Sim Numérico (ISO 4217) 3 (fixo) Sempre 986 — código ISO da moeda Real (BRL) 5303986
54 Transaction Amount Não Decimal string 1–13 (variável) Valor em reais com ponto decimal. Ex: 150.00. Se ausente, o pagador digita o valor. Nunca inclua separador de milhar. 5406150.00
58 Country Code Sim String (ISO 3166-1) 2 (fixo) Sempre BR 5802BR
59 Merchant Name Sim Texto livre 1–25 (variável) Nome do recebedor exibido ao pagador. Apenas o tamanho real do nome — sem padding. Caracteres: A-Z, 0-9, espaço, alguns especiais. 5913Fulano Silva
60 Merchant City Sim Texto livre 1–15 (variável) Cidade do recebedor. Tamanho real, sem completar com espaços. 6009SAO PAULO
62 Additional Data Field Não Estrutura aninhada variável Contém subcampo 05 (Reference Label = txid). Se não houver txid, colocar *** (obrigatório pelo padrão). O bloco 62 pode ser omitido apenas se não houver txid algum. 62070503*** ou 6229050125ABC123...
62-05 ↳ Reference Label (txid) Cond. Alfanumérico 1–25 (variável) Tamanho real do txid informado. Caracteres: a-z A-Z 0-9. Se sem txid, use *** (3 chars). Retransmitido intacto na ordem de pagamento. 050512ABC99
63 CRC16 Sim Hexadecimal 4 (fixo) Sempre o último campo. Algoritmo: CRC-16/CCITT-FALSE (poly 0x1021, init 0xFFFF). Calculado sobre todo o payload incluindo 6304. Resultado em maiúsculo, zero-padded. 63041D3D
80–99 Unreserved Templates Cond. Estrutura aninhada 23–99 (por bloco) Usado apenas no QR Code Composto (Pix Automático). Contém subcampos 00=GUI e 25=URL de recorrência. Obrigatório no Composto; proibido nos demais tipos. (apenas no QR Composto)
⚠️
O campo 63 (CRC) é a prova de integridade Todo QR Code Pix termina com um CRC16 de 4 caracteres. Se qualquer bit do payload for alterado, o CRC não bate e o pagamento é rejeitado. Isso protege contra adulterações em QR Codes impressos.

Exemplo de payload real (QR Estático)

Primeiro, a string real — exatamente como ela existe: um bloco contínuo sem separadores.

// String completa (é assim que o QR Code armazena — sem espaços, sem quebras): 00020101021226580014br.gov.bcb.pix0136123e4567-e12b-12d1-a456-4266554400005204000053039865802BR5913Loja Exemplo S6009SAO PAULO62070503***63041D3D

Agora, campo a campo — cada linha é um campo isolado:

// ─── campo 00 ────────────────────────────────────────────────────────────── 00 02 01 // ↑ ↑ ↑ // ID len valor // Payload Format Indicator — valor "01" é o único válido pelo EMV // ─── campo 01 ────────────────────────────────────────────────────────────── 01 02 12 // Point of Initiation Method // "11" = uso único (QR dinâmico) | "12" = reutilizável (QR estático) // ─── campo 26 (bloco Merchant Account Info) ──────────────────────────────── 26 58 ← abre o bloco (os próximos 58 chars são os subcampos abaixo) 00 14 br.gov.bcb.pix // GUI — identifica o arranjo Pix 01 36 123e4567-e12b-12d1-a456-426655440000 // Chave Pix (UUID, 36 chars com hifens) // ──── fim do bloco 26 ────────────────────────────────────────────────────── // ─── campo 52 ────────────────────────────────────────────────────────────── 52 04 0000 // Merchant Category Code (MCC) — "0000" para Pix (sem categoria específica) // ─── campo 53 ────────────────────────────────────────────────────────────── 53 03 986 // Transaction Currency — "986" = BRL (ISO 4217) // ─── campo 54 ────────────────────────────────────────────────────────────── // (ausente) → valor livre; o pagador digita o valor no app // ─── campo 58 ────────────────────────────────────────────────────────────── 58 02 BR // Country Code — "BR" (ISO 3166-1 alpha-2) // ─── campo 59 ────────────────────────────────────────────────────────────── 59 13 Loja Exemplo S // Merchant Name — 13 chars exatos, sem padding, sem espaço extra // ─── campo 60 ────────────────────────────────────────────────────────────── 60 09 SAO PAULO // Merchant City — 9 chars (sem acento — padrão EMV é ASCII) // ─── campo 62 (Additional Data Field Template) ───────────────────────────── 62 07 ← abre o bloco (7 chars de subcampos) 05 03 *** // Reference Label (txid) = "***" quando não há txid // ──── fim do bloco 62 ────────────────────────────────────────────────────── // ─── campo 63 ────────────────────────────────────────────────────────────── 63 04 1D3D // CRC-16/CCITT-FALSE calculado sobre TUDO acima + o próprio prefixo "6304" // Resultado em hexadecimal maiúsculo, sempre 4 chars
🔢
Como ler qualquer campo: ID (2) + Comprimento (2) + Valor 5913 → ID=59, len=13, valor = os próximos 13 chars. O comprimento é sempre o tamanho exato do valor — sem zeros de padding, sem espaços. Campo opcional vazio = campo inteiro ausente.

Os três tipos de QR Code Pix

O Manual de Padrões para Iniciação do Pix (v2.9.0) define três tipos. A diferença fundamental está em onde vivem os dados da cobrança:

Estático

QR Code Estático

Todos os dados estão dentro do próprio QR. Não precisa de servidor. Pode ser reutilizado infinitas vezes.

  • ✓ Geração offline
  • ✓ Múltiplos pagamentos
  • ✗ Sem valor automático por transação
  • ✗ Sem status de pagamento
📌 Exemplo real Uma padaria imprime um QR fixo no balcão. O cliente escaneia, digita R$ 7,50 pelo café e confirma. O mesmo QR funciona para o próximo cliente — sem expirar, sem backend.
payload BR Code (129 chars, CRC real):00020126470014br.gov.bcb.pix0125pagamentos@padaria.com.br5204000053039865802BR5915Padaria Ponto X6009SAO PAULO62070503***63044A17
Dinâmico

QR Code Dinâmico

O QR contém apenas uma URL. Os dados da cobrança são buscados nessa URL no momento da leitura.

  • ✓ Valor e txid únicos
  • ✓ Status em tempo real
  • ✓ Pode expirar
  • ✓ Cobrança com vencimento
📌 Exemplo real Um e-commerce gera um QR por pedido com valor fixo (R$ 349,90), txid único e validade de 30 min. Quando pago, o webhook notifica o sistema e o pedido é liberado automaticamente — sem intervenção humana.
payload BR Code (156 chars, CRC real):00020126550014br.gov.bcb.pix2533pix.loja.com.br/cobr/Ht7Kx9mNpQrT5204000053039865406349.905802BR5912Loja Exemplo6008CAMPINAS62200516Ht7Kx9mNpQrTwXzA630496CF
Composto

QR Code Composto

Extensão do dinâmico que adiciona uma segunda URL com parâmetros de recorrência — para o Pix Automático.

  • ✓ Adesão + pagamento num QR só
  • ✓ Base do Pix Automático
  • ✗ Não suporta Saque/Troco
📌 Exemplo real Uma academia oferece Pix Automático para mensalidades. O aluno escaneia o QR Composto uma única vez para aderir. Nos meses seguintes, o débito de R$ 120 acontece automaticamente no vencimento — sem novo escaneamento.
payload BR Code (218 chars, CRC real — note os dois campos 26):000201261130014br.gov.bcb.pix2533pix.academia.com.br/cobr/Rt5Mn8kL26540014br.gov.bcb.pix2532pix.academia.com.br/rec/Rt5Mn8kL5204000053039865406120.005802BR5915Academia FitPro6008CAMPINAS62200516Rt5Mn8kLpJhFcVbZ63046A11

Quando usar cada um?

CenárioTipo recomendadoMotivo
Cartão de visita, loja física com valor aberto Estático Reutilizável, simples, sem backend
E-commerce, maquininha, ERP Dinâmico (Cob) Valor único, conciliação automática via txid
Boleto digital, mensalidades, aluguel Dinâmico (CobV) Vencimento, juros, multa, desconto configuráveis
Assinatura, plano mensal, débito automático Composto Captura adesão ao Pix Automático no primeiro pagamento
🔁
Pix Copia e Cola é a versão texto do QR Code Tecnicamente, é o mesmo payload do QR Code — só que em formato texto para copiar e colar. O fluxo no app é idêntico. Suportado nos três tipos de QR.

Campos disponíveis

ID 26-51 > 00
GUI — br.gov.bcb.pix
Identificador do arranjo Pix. Obrigatório e fixo.
ID 26-51 > 01
Chave Pix
Chave válida no DICT. Obrigatório. Máx 77 chars.
ID 26-51 > 02
infoAdicional
Texto livre (ex: "Mesa 12"). Opcional. Máx 72 chars.
ID 26-51 > 03
fss
ISPB do facilitador de saque (Pix Saque). Opcional. 8 chars fixos.
ID 62-05
txid (Reference Label)
Identificador da transação para conciliação. Opcional no estático. Máx 25 chars.
ID 54
Valor
Valor pré-definido pelo recebedor. Opcional — se ausente, pagador define.
📏
Limite de 99 chars — a competição de espaço O bloco de dados Pix (IDs 26-51) tem no máximo 99 caracteres. chave + infoAdicional + fss dividem esse espaço:
  • Campo fss consome 12 chars fixos quando presente
  • Chave máxima sem outros campos: 77 chars
  • Quanto maior a chave (ex: e-mail longo), menos espaço para infoAdicional

Regras do txid no QR Estático

No QR Estático, o txid vai no campo EMV 62-05 (Reference Label):

RegraDetalhe
Tamanho máximo25 caracteres (diferente do dinâmico!)
Caracteres permitidosa-z, A-Z, 0-9
Sem txid definidoUse o valor *** (obrigatório pelo padrão)
TransmissãoPSP pagador deve retransmitir intacto na ordem de pagamento
OrigemCriação não faz parte da API Pix (gerado pelo recebedor)

Comparativo rápido

✅ QR Estático faz bem

  • Cobranças presenciais sem sistema
  • QR impresso em cardápio, balcão, pix.me
  • Múltiplos clientes, mesmo QR
  • Doações sem valor fixo

❌ QR Estático não faz

  • Conciliação confiável por pedido
  • Controle de expiração por transação
  • Bloquear segundo pagamento do mesmo QR
  • Cobrança com vencimento ou juros

QR Code Dinâmico — Cobrança Imediata (Cob)

O QR Code Dinâmico carrega uma URL no lugar dos dados da cobrança. Quando o app do pagador escaneia, ele faz uma requisição HTTPS para essa URL e obtém o payload JSON atualizado.

🔗
A URL não tem "https://" no QR Por spec, a URL é armazenada no BR Code sem o prefixo do protocolo. O app pagador adiciona automaticamente o https://. Isso economiza caracteres no QR Code.
Formato: pagamentos.banco.com.br/cobr/abc123xyz

O Payload JSON — estrutura completa da Cob

{ "revisao": 0, // incrementado a cada alteração via API "calendario": { "criacao": "2025-11-17T14:00:00Z", // RFC 3339 "apresentacao": "2025-11-17T14:35:12Z", // momento da leitura "expiracao": 3600 // segundos. Default=86400 (24h) }, "devedor": { "cpf": "12345678900", // ou cnpj, não ambos "nome": "Maria Oliveira" }, "valor": { "original": "150.00", "modalidadeAlteracao": 0 // 0=fixo, 1=pagador pode alterar }, "chave": "empresa@pagamentos.com.br", "txid": "ABC123DEF456GHI789JKL012MNO34", // 26-35 chars, único por CNPJ "solicitacaoPagador": "Ref. Pedido #9087", // max 140 chars → vai no RmtInf "infoAdicionais": [ { "nome": "Produto", "valor": "Monitor 27\"" }, { "nome": "SKU", "valor": "MON-4K-001" } ], "status": "ATIVA" // ATIVA | CONCLUIDA | REMOVIDO_PELO_USUARIO_RECEBEDOR | REMOVIDO_PELO_PSP }

Status possíveis da cobrança

ATIVA CONCLUIDA REMOVIDO_PELO_USUARIO_RECEBEDOR REMOVIDO_PELO_PSP
🔒
CONCLUIDA é status final e irreversível Uma vez que um Pix for confirmado, o status muda para CONCLUIDA e nenhum outro pagamento pode ser feito nesse QR. O txid é único por CPF/CNPJ — cada transação tem o seu.

Unicidade e o txid no Dinâmico

PropriedadeDinâmico (Cob/CobV)
TamanhoMínimo 26, máximo 35 caracteres
UnicidadeGarantida por CPF/CNPJ do recebedor (pelo PSP recebedor)
GeraçãoPelo recebedor, ou delegada ao PSP (só em Cob)
TransmissãoRetransmitido intacto pelo PSP pagador na mensagem ISO 20022
Nos campos do QRNão vai no BR Code — está só no payload JSON da URL

Cobrança com Vencimento (CobV)

A CobV é o "boleto do Pix" — uma cobrança com data de vencimento, podendo conter juros por atraso, multa, desconto para pagamento antecipado e abatimento. O QR Code é obrigatoriamente dinâmico.

📅
Base legal Art. 11-A, II da Res. BCB 1/2020 (incluído pela Res. BCB 88/2021). A CobV pode ser paga após o vencimento se o recebedor configurar o campo validadeAposVencimento.

Campos adicionais da CobV (sobre a Cob)

// Campos adicionais no calendario: "calendario": { // ... mesmos campos da Cob, mais: "dataDeVencimento": "2025-12-20", // formato yyyy-mm-dd "validadeAposVencimento": 30 // dias corridos após vencimento }, // Devedor com campos extras disponíveis na CobV: "devedor": { "cpf": "12345678900", "nome": "João Silva", "logradouro": "Rua das Flores, 100", "cidade": "Campinas", "uf": "SP", "cep": "13000000" }, // Configurações financeiras: "valor": { "original": "500.00", "abatimento": { "modalidade": 1, // 1=valor fixo, 2=percentual "valorPerc": "10.00" // R$10 de abatimento fixo }, "desconto": { "modalidade": 1, // 1=desconto por data fixa "descontoDataFixa": [ { "data": "2025-12-15", "valorPerc": "20.00" }, // R$20 até 15/12 { "data": "2025-12-18", "valorPerc": "10.00" } // R$10 até 18/12 ] }, "juros": { "modalidade": 2, // 2=percentual ao dia "valorPerc": "0.033" // 0,033% ao dia de atraso }, "multa": { "modalidade": 2, // 2=percentual sobre valor original "valorPerc": "2.00" // 2% de multa } }

Vf = Vo − Va − Vd + Vj + Vm

VaAbatimentoSempre (se configurado)
VdDescontoAntes ou na data de vencimento
VjJuros por atrasoApós vencimento
VmMultaApós vencimento (aplicada uma vez)

DICT e Chaves Pix — o diretório central

O DICT (Diretório de Identificadores de Contas Transacionais) é o coração do Pix. É o sistema do Banco Central que mapeia chaves Pix para contas bancárias. Sem ele, o Pix não funciona.

🏛️
Por que o DICT é fundamental para o QR Code? Todo QR Code Pix (estático ou dinâmico) tem uma chave no payload. Quando o app do pagador lê o QR, a primeira coisa que faz é consultar o DICT com essa chave para descobrir: qual banco? Qual agência? Qual conta?

Fluxo completo de pagamento por QR Code

Vamos agora juntar tudo. Quando você aponta a câmera para um QR Code Pix, o que acontece nos bastidores?

Fluxo — QR Code Estático

1
Pagador abre app e acessa "Pix > Ler QR Code"

Obrigatório por regulação estar na tela principal do Pix.

2
App lê o QR Code e decodifica o payload BR Code

Extrai o campo da chave (ID 26-51 > 01) e o valor, se houver (ID 54).

3
PSP Pagador consulta o DICT com a chave encontrada

Envia EndToEndId da transação futura e CPF/CNPJ do pagador. DICT retorna dados do recebedor.

4
App exibe tela de confirmação ao pagador

Nome do recebedor, CPF/CNPJ mascarado, PSP, valor (editável se livre).

5
Pagador confirma → PSP Pagador envia pacs.008 ao SPI

Mensagem ISO 20022 com todos os dados. Liquidação final no SPI (Sistema de Pagamentos Instantâneos) do Banco Central.

6
SPI liquida e PSP Recebedor notifica usuário

Crédito disponível em até 10 segundos. PSP recebedor pode usar webhook da API Pix para ser notificado.

Fluxo — QR Code Dinâmico (Cob/CobV)

1
Recebedor cria cobrança via API Pix (POST /cob ou /cobv)

Para CobV: exibe vencimento, descontos, juros/multa e valor final calculado.

Mesmo mecanismo do estático. PSP recebedor recebe notificação via webhook com o txid.

PSP recebedor atualiza status para CONCLUIDA

Cobrança vinculada ao pagamento via txid. Conciliação automática no ERP do recebedor.

A API Pix é o que permite ao recebedor (empresa, fintech, desenvolvedor) automatizar toda a gestão de cobranças. É uma API REST com autenticação OAuth 2.0, disponível em: github.com/bacen/pix-api

🔄

Segurança e mecanismos de proteção

🔐
CRC16 — integridade do QR Code

Todo QR Code termina com 4 chars de CRC16. Qualquer adulteração no payload (troca de chave, mudança de valor) invalida o checksum e o pagamento é recusado automaticamente.

🌐
HTTPS obrigatório para QR Dinâmico

A URL do QR Dinâmico deve ser acessada exclusivamente via HTTPS com domínio válido. O app adiciona automaticamente o prefixo — não há como servir payload via HTTP.

🚫
Sem renderização HTML

O campo solicitacaoPagador (a "mensagem" do recebedor) é exibido como texto plano. Apps não devem renderizar HTML. Isso previne ataques de phishing via QR Code.

🪣
Token Bucket no DICT

Para evitar ataques de leitura em massa (varredura do DICT para mapear CPFs/chaves), o DICT usa algoritmo Token Bucket. PF: 100 fichas. PJ: 1.000 fichas. Consulta inválida consome 20 fichas. Receber um Pix reabastece o balde.

🔍
MED — Mecanismo Especial de Devolução

Quando um usuário é vítima de golpe, seu PSP pode acionar a "Recuperação de Valores" no DICT. O sistema rastreia o grafo de transações subsequentes e pode bloquear/devolver valores em cascata, mesmo que o dinheiro já tenha sido movimentado várias vezes.

📊
Informações de segurança por chave

PSPs podem consultar o DICT para saber o histórico de fraudes de uma chave: notificações confirmadas, tipo de fraude (conta laranja, falsidade ideológica, golpista), valores envolvidos — nos últimos 90 dias, 12 meses e 60 meses.

Resumo executivo — o que guardar

🎯 As 7 ideias que não podem sair da cabeça

  1. BR Code = EMV adaptado ao Brasil. O campo GUI br.gov.bcb.pix é a assinatura do Pix.
  2. Estático = dados no QR. Dinâmico = URL no QR. A diferença muda tudo: conciliação, expiração, controle.
  3. Todo QR tem uma chave. O app sempre consulta o DICT com essa chave para resolver banco/conta.
  4. CobV é o boleto do Pix. Juros, multa, desconto, vencimento — tudo configurável via API Pix.
  5. txid é o ID de conciliação. No estático: 25 chars no campo 62-05. No dinâmico: 26–35 chars no payload JSON.
  6. Webhook = eficiência. Sem polling. O PSP notifica o recebedor quando o Pix cai, com o txid para reconciliar automaticamente.
  7. UX não é opcional. O BCB define campos obrigatórios na tela de confirmação. PSP que não exibe é PSP irregular.

Seção Técnica — Implementação para Desenvolvedores

🧑‍💻
Para quem é essa seção Engenheiros e arquitetos implementando Pix QR Code dentro de um internet banking (PSP). Cobre stack de mensageria, geração do BR Code, API Pix server/client, segurança obrigatória e anti-fraude. Baseado em Res. BCB 195/2022 (SPI), Res. BCB 85/2021 (segurança), Res. BCB 403/2024 (dispositivos) e Catálogo de Serviços do SFN v5.12.

T1. Primeiro passo: participação no SPI

Antes de qualquer linha de código, a instituição precisa ser participante do SPI — o sistema de liquidação do BCB. Há duas modalidades (Res. BCB 195/2022, Art. 14):

Participante Direto

  • Titular de Conta PI no BCB
  • Conexão direta à RSFN
  • Bancos comerciais, múltiplos, caixas, cooperativas de crédito
  • Responsabilidade total pela liquidação
  • Deve estar conectado 24/7/365 (Art. 18, II)

Participante Indireto

  • Sem Conta PI própria
  • Opera via um banco liquidante (direto)
  • IPs sem autorização BCB são obrigados a ser indiretos
  • Conexão com o liquidante é livre (contratual)
  • Vedado para bancos comerciais e múltiplos
📌
PSTI — Provedor de Serviços de TI A conexão à RSFN pode ser feita diretamente via circuito de telecom, ou via um PSTI (Provedor de Serviços de Tecnologia da Informação) autorizado pelo BCB. O PSTI cuida da infraestrutura de rede, mas a responsabilidade normativa permanece com o participante.

T2. Stack de mensageria — RSFN + ISO 20022

O Pix trafega na RSFN (Rede do Sistema Financeiro Nacional) — uma rede dedicada, segregada da internet pública. As mensagens seguem o padrão ISO 20022 em XML.

Estrutura de toda mensagem RSFN

// Toda mensagem RSFN é composta por três camadas: BCMSG (≤ 2.768 bytes) // Cabeçalho de controle de transporte └─ NumSeq // Sequência para mensagens em partes └─ IndUltMsg = 'S' // 'S' na última parte SISMSG (≤ 30.000 bytes) // Envelope do sistema SFN USERMSG (≤ 1 MB) // Payload de negócio — aqui vive o ISO 20022 └─ pacs.008 // Ordem de pagamento (Pix de saída) └─ pacs.004 // Devolução └─ pacs.002 // Status / confirmação // Codificação: UTF-8 para mensagens SPI (LPI group)

T3. End-to-End ID (E2EID) — formato obrigatório

Toda transação Pix tem um E2EID único, gerado pelo PSP emissor. É o identificador que viaja na mensagem pacs.008 e permite rastrear a transação no SPI.

// Formato: 32 caracteres alfanuméricos // E + [8 ISPB do emissor] + [8 data yyyyMMdd] + [4 hora HHmm] + [11 aleatórios] E12345678202511171435PRH1a2B3c4D │ │ │ │ │ │ │ │ │ └─ 11 chars aleatórios (a-z, A-Z, 0-9) │ │ │ └────── Hora UTC (HHmm) │ │ └─────────────── Data UTC (yyyyMMdd) │ └──────────────────────── ISPB do participante emissor (8 dígitos) └────────────────────────── Letra 'E' (fixo) // Exemplo real: "E12345678202511171435PRH1a2B3c4D" // 32 chars // Atenção: horário sempre em UTC (não horário local de Brasília) // O SPI usa UTC como padrão — Art. 10 Res. BCB 195/2022

T4. Gerando o payload do BR Code (QR Estático)

O payload é uma string de texto concatenando campos no formato ID (2 chars) + Comprimento (2 chars) + Valor, finalizando com CRC16.

// Função de geração em pseudocódigo: function field(id, value): length = zeroPad(value.length, 2) return id + length + value // Montagem do payload: payload = field("00", "01") // Payload Format Indicator payload += field("26", // Merchant Account Info field("00", "br.gov.bcb.pix") + field("01", chave) // Chave Pix ) payload += field("52", "0000") // MCC payload += field("53", "986") // Moeda BRL if valor: payload += field("54", formatDecimal(valor)) // ex: "150.00" payload += field("58", "BR") // País payload += field("59", nome) // Nome recebedor (max 25) payload += field("60", cidade) // Cidade (max 15) payload += field("62", field("05", txid || "***")) // txid // CRC16 — SEMPRE o último campo: payload += "6304" // Prepara campo 63 com "0000" crc = crc16_ccitt(payload) // Calcula sobre tudo incluindo "6304" payload += zeroPadHex(crc, 4).toUpperCase() // ex: "1D3D"
⚙️
CRC16 — algoritmo exato Usar CRC-16/CCITT-FALSE (polinômio 0x1021, valor inicial 0xFFFF, sem inversão de bits). O campo 63 deve ser incluído no cálculo com o valor 6304 (os 4 zeros do hash ainda não calculado). O resultado vai em hexadecimal maiúsculo, 4 chars, preenchido com zeros à esquerda.

Para QR Dinâmico — a URL substitui os dados

// No dinâmico, o campo 01 da faixa 26 é a URL sem https:// // A URL aponta para o endpoint de cobrança na API Pix do PSP recebedor: payload += field("26", field("00", "br.gov.bcb.pix") + field("25", "api.banco.com.br/cobr/Hj2Kx9mNpQrT") // ← URL sem https:// ) payload += field("52", "0000") payload += field("53", "986") // NÃO inclui campo 54 (valor) — vem da cobrança, não do QR payload += field("58", "BR") payload += field("59", "Loja Exemplo") payload += field("60", "SAO PAULO") payload += field("62", field("05", "***")) // txid pode ser "***" no dinâmico payload += "6304" crc = crc16_ccitt(payload) payload += zeroPadHex(crc, 4).toUpperCase()
💡
QR Dinâmico não carrega valor no payload do QR O valor está na cobrança (Cob/CobV) que a URL retorna via API Pix. O app do pagador lê o QR, faz GET na URL, recebe o JSON com valor, vencimento e dados do recebedor — e só então exibe a tela de confirmação.

T6. Webhook — recebendo notificação de pagamento

Ao invés de pollear a API, configure um webhook para receber POST do PSP assim que o Pix for liquidado. O endpoint deve estar acessível publicamente com TLS válido.

PUT /webhook/{chave} — cadastrar URL de callback

// Registra a URL que vai receber os POSTs: { "webhookUrl": "https://api.suaempresa.com.br/pix/webhook" } // Payload que você vai RECEBER (POST do PSP): { "pix": [ { "endToEndId": "E1234567820251117143500abc123xyz", "txid": "Hj2Kx9mNpQrTwXzAbCdEfGh", "valor": "150.00", "horario": "2025-11-17T14:35:00.000Z", "pagador": { "cpf": "12345678909", "nome": "João Silva" } } ] }
⚠️
Idempotência no webhook O PSP pode re-enviar o mesmo evento. Sempre verifique se o endToEndId já foi processado antes de executar ações (liberar produto, acionar processo downstream). Armazene E2EIDs processados.

T10. Devoluções via API e MED

Existem dois mecanismos de devolução: a devolução simples via API (solicitada pelo recebedor ou pagador) e o MED (Mecanismo Especial de Devolução), acionado pelo BCB em casos de fraude ou falha operacional.

TipoEndpointPrazoQuem aciona

T12. Tratamento de erros e SLA

A API Pix usa códigos HTTP padrão mais um campo type (URI) para categorizar erros. O SLA de disponibilidade e tempo de resposta é regulado pela Res. BCB 85/2021.

HTTPtypeSignificado
400https://pix.bcb.gov.br/api/v2/error/CobOperacaoInvalidaParâmetros inválidos na cobrança
401.../NaoAutorizadoToken expirado ou escopo insuficiente
403.../AcessoNegadoChave não pertence ao PSP
404.../CobNaoEncontradatxid não existe
503.../ServicoIndisponivelSPI em janela de manutenção ou sobrecarga
📋
SLA regulatório (Res. BCB 85/2021, Art. 6) Disponibilidade mínima de 99,5% medida mensalmente. Tempo máximo de resposta para liquidação: 10 segundos em condições normais. Janelas de manutenção programada devem ser comunicadas ao BCB com antecedência mínima de 72h.
